System Architecture

The booking system has three layers: Wodify (class schedule), Google Apps Script (backend logic), and the website frontend (booking pages). Staff manage class schedules exclusively in Wodify — no Google Calendar maintenance required.

Data Flow Summary
  1. Wodify API → PropertiesService cache (daily 3 am)
  2. PropertiesService + CFR Bookings sheet → CacheService (every 4 h)
  3. Booking page requests → Apps Script → CacheService (fast, no API call)
  4. Customer pays → Square API → Sheets write → Wodify lead → Confirmation email
What Staff Need to Know
  • Manage classes in Wodify only — no Google Calendar
  • Google Calendar (Reservations) is auto-written for staff reference — don't edit it
  • After adding/changing Wodify classes: run updateWodifyCache
  • Cache preloads automatically every 4 h via scheduled trigger

Managing Classes in Wodify

✅
Wodify is the single source of truth for the class schedule. Do not edit Google Calendar manually. The website reads the schedule exclusively from Wodify via the API cache.

Wodify Program Names — Must Be Exact

The website maps Wodify program names to class types using an exact string match. The program names in Wodify admin must match these exactly (case-sensitive):

Wodify Program NameMaps To (website)Status
CrossFitDrop InActive
Open GymOpen GymActive
HYROX ClassHYROX PerformanceActive
Strength ClassHYROX StrengthActive
TrialTrial ClassCreate in Wodify admin
SpartanSpartanCreate or confirm in Wodify admin
FoundationFoundationCreate in Wodify admin
⚠️
Ignored programs — these will NOT appear on the website: CrossFit Mobility Class, Gymnastics Class, Competitors Class, Olympic Lifting, Games Open. You can still create/run these in Wodify — the website simply ignores them.

Workflow: Adding or Editing a Class

  1. In Wodify admin — create or edit the class under the correct program. Set the date, time, and recurring schedule as needed.
  2. Run cache update — open the Apps Script Web App URL with ?action=updateWodifyCache. This takes 5–10 seconds and scans the most recent Wodify pages to pick up changes.
    [GAS_URL]?action=updateWodifyCache
  3. Verify — run ?action=listWodifyClasses&date=YYYY-MM-DD to confirm the class appears with the correct program.
  4. Clear availability cache (optional) — the booking pages cache availability for 4 hours. If you need the change to appear immediately, run ?action=cacheStatus and note that a new _warmCache() run will refresh everything automatically within 4 hours.

Workflow: Deleting or Cancelling a Class

  1. Cancel/delete in Wodify admin.
  2. Run ?action=buildWodifyCache — the incremental update only adds new entries. To remove a deleted class, a full rebuild is needed. This takes 3–5 minutes.
💡
Full rebuild vs. incremental update: updateWodifyCache is fast (~5 sec) and handles new classes. Use it for additions. buildWodifyCache scans all 400+ pages of Wodify history (~3–5 min) and is needed only when classes are deleted or the cache is stale/corrupt.

Drop In 3-Pack Credits

Purchase

Customers buy the 3-pack during Drop In booking (Step 3 upsell). They pay ¥11,550 and receive 3 credits in the CFR Credits sheet tab. Expiry: 90 days from purchase date.

Redemption at Front Desk

Open admin-credits.html (password protected). Enter customer email or phone to look up their credits. Select a class slot, click Use Credit. Apps Script logs the redemption and decrements the credit count.

Credit Rules

  • Valid for Drop In and Open Gym only
  • Expire 90 days from purchase
  • Expire whether used or not — non-extendable
  • 1 credit = 1 class (any Drop In/Open Gym slot)
  • Front desk redemption logged as source: front_desk in CFR Bookings sheet
⚠️
Checking customer credits: Use the GET action ?action=checkCredits&email=CUSTOMER_EMAIL (or phone) to see remaining credits and expiry before redemption.
⚙️

Scheduled Triggers

Apps Script triggers run automatically. They are set up via _setupAllTriggers() and do not require manual action.

FunctionSchedulePurpose
_updateWodifyClassCache Daily at 3:00 am JST Pull latest class data from Wodify API and update PropertiesService cache
_warmCache Every 4 hours Precompute availability for next 14 days (dropin/opengym/trial) + upcoming classes (hyrox/spartan/foundation). Stores results in CacheService.
_sendReminders Daily at 8:00 am JST Send reminder emails to Trial and Foundation customers booked for tomorrow
💡
Cache warm-up timing: The Wodify cache refreshes at 3 am. The availability CacheService then refreshes at 4 am, 8 am, 12 pm, 4 pm, and 8 pm (every 4 hours). The website is never more than 4 hours stale. For immediate updates, run updateWodifyCache manually, which triggers a fresh read on the next request.

Google Sheets

The spreadsheet has 4 tabs. Apps Script writes to all of them; do not edit rows that have booking data or you may corrupt the records.

TabPurposeWritten by
CFR Bookings All class bookings — one row per booking. Column AB (index 27) = confirmed / cancelled. Columns G/H (index 6/7) = class date/time used for booking counts. _postCreateBooking, _postUseCreditFrontDesk
CFR Forms Contact form submissions from contact.html _postSubmitContactForm
CFR Careers Job application submissions from career.html _postSubmitCareerForm
CFR Credits Drop In 3-pack credit balances. Each row = one purchase. Credits decremented on redemption. _postCreateBooking (purchase), _postUseCreditFrontDesk (redeem)
⚠️
Do not modify column order or add columns between existing columns in CFR Bookings. The Apps Script reads by column index, not header name. Adding a column in the middle will silently corrupt all subsequent reads.

Troubleshooting

Class not appearing on website
  1. Verify the class exists in Wodify admin under the correct program (check exact name spelling).
  2. Run ?action=updateWodifyCache and check the response shows "new_entries": N where N > 0.
  3. Run ?action=listWodifyClasses&date=YYYY-MM-DD to confirm the class appears in the cache with correct program.
  4. Run ?action=availability&date=YYYY-MM-DD&classType=dropin to confirm it appears in the API response.
  5. If still missing after cache update: check the program name in Wodify exactly matches the WODIFY_CLASS_MAP keys (case-sensitive).
Class shows as full but it shouldn't be
  1. Run ?action=availability&date=YYYY-MM-DD&classType=dropin — check the booked count in the response.
  2. Open CFR Bookings sheet and filter for that date + class type. Count rows with status = confirmed.
  3. If the sheet count is lower than the API booked count, the CacheService may have stale data. Wait up to 4 hours for _warmCache() to refresh, or restart by running ?action=availability directly (bypasses cache on first miss).
Cache returns 0 entries after buildWodifyCache

This is expected if the cache build was interrupted early. Wodify orders classes by creation time — future classes are on the last pages (459–461 of ~461 pages). The build must complete fully.

Use ?action=updateWodifyCache instead — it scans the last 3 pages and takes ~5 seconds.

Payment succeeded but no confirmation email
  1. Check CFR Bookings sheet — does a row exist? If yes, the booking was saved. The email send may have failed independently.
  2. Check Apps Script execution logs in the GAS IDE (Executions panel) for GmailApp errors.
  3. Check GmailApp daily quota — Apps Script has a limit of ~100 emails/day on consumer accounts. Workspace accounts have a higher limit.
  4. Forward the booking confirmation manually from bookings@crossfitroppongi.com if needed.
Booking created with wrong Wodify placeholder client

The Wodify placeholder clients (WODIFY_TRIAL_CLIENT_IDS, WODIFY_DROPIN_CLIENT_IDS) are assigned based on how many confirmed bookings already exist for that slot. If the count is off, check CFR Bookings for duplicate or incorrect status values for that date/time.

Apps Script returns "Server error" or crashes
  1. Open Google Apps Script IDE → Executions panel to see the full error stack trace.
  2. Common causes: Square API timeout, Sheets API quota exceeded (short burst), Wodify API key expired, or PropertiesService size limit hit (50 KB max).
  3. If PropertiesService is full, run ?action=buildWodifyCache — it rebuilds fresh and will fail if cache exceeds 50 KB. In that case, trim ignored programs from WODIFY_CLASS_MAP or reduce cache window.
